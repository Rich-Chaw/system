## 摘要

网络鲁棒性评估与网络拆解（Network Dismantling）是复杂网络分析的重要方向，其核心目标是在尽可能少的节点移除代价下，使网络最大连通分量（Largest Connected Component, LCC）快速崩解，从而揭示网络脆弱点并为关键节点干预提供决策依据。本文面向“网络拆解系统”的工程实现，设计并实现了一套面向Web的网络拆解与模型评估原型系统（NetworkDismantling），采用“前端组件化 + Flask服务端 + 多模型执行器”的三层结构，支持图数据上传/生成、拆解评估、多模型对比、可视化与结果导出；同时系统提供基础的用户鉴权与管理员控制台能力。针对毕业设计的工程完整性要求，本文进一步提出面向生产化的功能扩展方案，包括个人信息页与用户历史记录、模型管理、训练记录持久化、断点续训、存储空间监控、数据集管理、训练任务监控与日志CSV导出，并给出SQLite数据层设计、关键业务流程、模块划分与测试方案。

关键词：网络拆解；复杂网络；强化学习；Flask；前后端分离；模型管理；训练任务；断点续训；SQLite

## Abstract

Network dismantling aims to disrupt the largest connected component (LCC) of a graph by removing as few nodes as possible, enabling robustness analysis and critical-node intervention in complex networks. This report presents a web-based prototype system (NetworkDismantling) for network dismantling and model evaluation. The system adopts a three-layer architecture: a component-based frontend, a Flask backend, and an isolated multi-model execution layer via conda subprocesses. It supports graph upload/generation, dismantling evaluation, multi-model comparison, visualization, and result export, along with basic authentication and an admin console. To meet software engineering graduation requirements, this report further proposes production-oriented extensions, including user profiles and histories, model management, persistent training records, resume-from-checkpoint, storage monitoring, dataset management, training task monitoring, and CSV export of operation logs, with a concrete SQLite schema, core workflows, module design, and test plan.

Keywords: network dismantling; complex networks; reinforcement learning; Flask; web system; model management; checkpoint resume; SQLite

## 第1章 绪论

### 1.1 研究背景与意义

复杂网络普遍存在于社会网络、交通网络、通信网络与生物网络等领域。网络鲁棒性研究关注网络在遭受攻击或故障下的连通性变化。网络拆解问题以LCC随节点移除的演化为核心度量，通过优化移除序列以最小化鲁棒性面积，从而识别关键节点与脆弱结构。近年来，基于深度强化学习的策略模型（如GraphDQN类方法）能够在不同网络结构上学习通用性拆解策略，为工程化应用提供可能。

然而，学术模型从代码到可用系统之间仍存在显著鸿沟：图数据获取与校验、模型运行环境隔离、长任务执行与监控、结果可视化、可重复实验、模型/数据版本管理、权限与审计等均需要系统化的软件工程支撑。因此，构建一套结构清晰、可扩展且可演示的网络拆解系统具有工程与教学意义。

### 1.2 研究内容与主要工作

本文的主要工作包括：
- 设计并实现网络拆解评估原型系统：支持图上传/生成、模型选择、拆解执行、多模型对比与可视化。
- 设计统一的多模型执行框架：以接口脚本为边界，通过conda环境隔离运行不同项目的模型，统一输入输出协议。
- 设计基础的用户体系与管理员能力：登录/注册、权限区分、用户管理、操作日志与系统统计。
- 在原型基础上提出扩展功能方案：个人信息页、模型管理、训练记录持久化、断点续训、存储监控、数据集管理、训练任务监控与日志CSV导出，并形成可落地的数据层与业务层设计。

## 第2章 相关技术与系统现状

### 2.1 相关技术

#### 2.1.1 前端技术

系统前端采用原生JavaScript（ES6+）与Bootstrap 5构建页面交互与布局，并使用D3.js/Chart.js完成图结构与指标曲线的可视化。前端以组件化思路实现复用，通过EventBus实现跨组件事件通信。

#### 2.1.2 后端技术

后端采用Flask提供REST API，使用flask-cors处理跨域访问。图处理依赖NetworkX，部分模型（如MIND-ND与baseline）使用igraph格式，系统提供NetworkX与igraph之间的格式转换与属性补齐。

#### 2.1.3 多模型执行与环境隔离

为了在同一系统中运行来自不同项目栈的模型（TensorFlow/PyTorch等），系统采用“接口脚本 + conda run”的方式隔离依赖环境：后端将图序列化为临时pkl文件，调用对应模型接口脚本执行，并从结果json文件中读取拆解序列、鲁棒性与LCC演化等指标，形成统一返回格式。

### 2.2 现有系统功能基线（实现现状）

本节基于现有目录 `system/` 的实现与架构文档，对系统已实现能力进行归纳，作为后续扩展设计的基线。

#### 2.2.1 系统角色与权限（现状）

系统当前定义两类角色：
- 普通用户（user）：可进行模型评估与训练页面访问（训练页当前为前端模拟）、可登录/退出。
- 超级管理员（superadmin）：可访问管理控制台，执行用户管理与日志查看。

系统使用token机制进行页面访问控制：前端将token保存在LocalStorage，并通过HTTP Header `X-Auth-Token`传递给后端。管理员权限通过后端校验token对应用户角色为`superadmin`实现。

#### 2.2.2 评估功能（模型拆解）现状

评估页面提供：
- 图数据输入：上传文件、手动边列表、预设图生成（BA/ER）。
- 模型配置：选择模型与参数（例如step ratio、最大迭代等）。
- 拆解执行：调用后端拆解接口并返回移除序列、鲁棒性、LCC演化及用于可视化的逐步状态（step-by-step）。
- 结果导出：导出拆解方案与JSON结果。

#### 2.2.3 训练功能现状

训练页面提供超参数输入、训练曲线展示与“检查点列表”，但当前实现为前端定时器模拟训练过程，并未形成真实后端训练任务、持久化记录与断点续训能力。该部分将在扩展功能中进行工程化补齐。

#### 2.2.4 管理控制台现状

管理员控制台提供：
- 系统统计：用户数、会话数、模型数、服务器运行状态。
- 用户管理：新增用户、禁用/启用用户、删除用户。
- 操作日志：查看近期操作日志（如登录、注册、用户管理操作）。

需要指出：当前用户、会话与操作日志数据存放于后端内存结构中，属于演示原型实现；本文扩展设计将给出SQLite持久化与可导出机制。

#### 2.2.5 后端接口现状（概览）

现有API大致分为四类：
- 公共/基础：健康检查、模型列表
- 图相关：上传图、生成预设图
- 拆解评估：单模型拆解、多模型拆解、方案评估
- 鉴权与管理：登录/注册/退出/me；管理员用户管理与日志；系统统计

### 2.3 现状问题与工程化改进动机

结合毕业设计“可复现、可管理、可扩展”的系统工程要求，现状仍存在以下不足：
- 数据层缺失：用户、操作日志、拆解记录、训练记录等均缺少可查询的持久化存储。
- 训练链路未打通：训练任务未在后端执行，缺少任务状态、曲线与checkpoint的持久化与断点续训。
- 模型治理缺失：模型元信息（来源、版本、文件占用、性能指标）未统一管理，缺少重命名/删除/下载与审计。
- 存储与数据集管理不足：缺少数据集复用机制与磁盘用量监控/清理策略。
- 管理侧可观测性不足：缺少跨用户训练任务监控与日志CSV导出。

## 第3章 需求分析

### 3.1 需求获取与分析方法

需求来源包括：系统现状功能清单、毕业论文工程要求、用户（实验人员/管理员）操作流程、以及新增功能列表。本文采用用例驱动与分层分模块分析方法，输出可追踪的功能性需求（FR）与非功能性需求（NFR），并对关键业务流程给出约束与验收标准。

### 3.2 系统角色定义

#### 3.2.1 普通用户

普通用户主要目标为：上传/选择图数据，选择模型进行拆解评估，查看结果与可视化；配置并启动训练任务，查看训练曲线与历史记录；管理个人资料（邮箱/头像/密码）。

#### 3.2.2 超级管理员

超级管理员除普通用户能力外，还需要：管理用户、审计操作日志、监控系统存储与训练任务，进行模型/数据集的清理与维护，导出日志CSV等。

### 3.3 功能性需求（FR）—新增功能清单与可追踪条目

本节在现有能力基础上扩展需求，并为后续设计与测试提供编号化追踪。

#### FR-01 个人信息页（`profile.html`）

- **FR-01-1**：用户可查看与编辑个人信息：邮箱、头像（上传/更换/删除）。
- **FR-01-2**：用户可修改密码（需校验旧密码与密码复杂度策略）。
- **FR-01-3**：用户可查看个人操作历史：拆解记录与训练记录的列表、筛选与详情查看。
- **FR-01-4**：所有个人信息修改操作应写入操作日志（可审计）。

#### FR-02 模型管理

- **FR-02-1**：模型列表页展示系统已保存模型（按类型：finder/mind/baseline；或统一“模型资产”视角）。  
- **FR-02-2**：支持模型重命名（只影响系统内展示名与元数据，不破坏文件引用）。
- **FR-02-3**：支持模型删除（按权限：普通用户仅可删除自己训练产物；管理员可删除任意模型；删除需审计）。
- **FR-02-4**：支持模型下载（HTTP文件下载，包含校验与权限）。

#### FR-03 训练记录详情持久化

- **FR-03-1**：训练任务创建时持久化训练参数（超参数、图分布配置、实验名、创建者等）。
- **FR-03-2**：训练过程中持久化训练曲线数据（按epoch/step记录reward、loss、lcc等）。
- **FR-03-3**：训练完成后持久化性能指标（最优指标、最终指标、耗时、模型文件路径与大小等）。
- **FR-03-4**：支持按任务/模型查看训练详情：参数、曲线、checkpoint列表、最终产物与评估指标。

#### FR-04 训练断点续训

- **FR-04-1**：训练任务支持生成checkpoint（按保存间隔或最优策略）。
- **FR-04-2**：支持从指定checkpoint恢复训练（resume），并保证训练曲线与指标连续性（或注明分段策略）。
- **FR-04-3**：支持训练任务的暂停/继续/停止（状态可追踪、可恢复、可审计）。

#### FR-05 存储空间监控与清理

- **FR-05-1**：显示磁盘用量概览（总量/已用/剩余）与主要目录占用（模型、数据集、日志、临时文件）。
- **FR-05-2**：管理员可执行清理操作（例如清理过期临时文件、删除已标记删除的模型文件、清理无引用数据集）。
- **FR-05-3**：清理操作需权限控制与操作日志记录。

#### FR-06 数据集管理

- **FR-06-1**：支持上传与管理常用图数据集（文件/元数据：名称、格式、节点边规模、来源、版本）。
- **FR-06-2**：训练与评估支持从已上传数据集中选择输入（复用）。
- **FR-06-3**：数据集权限：普通用户可管理自己上传的数据集；管理员可管理全局数据集。

#### FR-07 训练任务监控（管理员）

- **FR-07-1**：管理员可查看所有用户训练任务的状态列表（running/paused/failed/succeeded等）。
- **FR-07-2**：支持查看任务详情（参数、日志、曲线、checkpoint、资源占用摘要）。
- **FR-07-3**：支持对异常任务执行管理动作（停止/标记失败/清理产物；按策略）。

#### FR-08 系统日志导出（CSV）

- **FR-08-1**：管理员可按时间范围、用户名、动作类型过滤操作日志。
- **FR-08-2**：支持导出CSV下载（UTF-8 with BOM或明确编码策略），字段包含：时间、用户、动作、详情、IP/UA（如采集）、结果（成功/失败）等。
- **FR-08-3**：导出行为本身也需写入操作日志。

### 3.4 非功能性需求（NFR）

#### NFR-01 安全性
- 鉴权：所有受保护接口必须校验token有效性；管理员接口必须校验角色。
- 授权：模型/数据集/训练任务等资源需基于所有者与角色授权。
- 审计：用户关键操作（登录、资料修改、模型删除、清理、导出）均需记录审计日志。

#### NFR-02 性能与并发
- 评估接口属于“计算型长请求”，需限制超时并提供队列化/异步化的扩展路径。
- 训练任务属于“长时后台任务”，需从同步HTTP剥离，采用任务表驱动的后台执行器（未来可扩展到Celery/队列）。

#### NFR-03 可靠性与可恢复性
- 训练断点续训需保证checkpoint文件与数据库元信息一致；支持任务异常恢复与失败诊断。
- 清理策略需避免误删：建议引入软删除或回收站机制。

#### NFR-04 可维护性与可扩展性
- 采用三层分离：表现层、业务层、数据层；业务逻辑封装为services，便于测试与扩展。
- 模型执行器采用“接口脚本协议”解耦不同项目，便于新增模型类型。

#### NFR-05 可用性与可解释性
- 关键页面提供明确的状态提示（训练状态、任务进度、错误原因）。
- 评估结果提供指标解释与可视化（LCC演化曲线、鲁棒性、移除序列）。

### 3.5 约束与假设（本报告默认）

- **数据持久化默认采用SQLite**：满足单机部署、毕业答辩演示与可复制性要求。
- **文件资产采用目录化存储**：模型文件、数据集文件、训练日志与checkpoint均保存在受控目录，并在DB中保存索引。
- **训练执行机制**：采用后台执行器（进程/线程）+ 任务表轮询的方式实现，便于与现有Flask架构兼容；未来可替换为队列/分布式方案。

## 第4章 总体设计

### 4.1 系统总体架构

系统采用三层架构并包含独立的“模型执行环境层”：
- 表现层（Client）：多页面（评估、训练、管理、个人信息、模型/数据集/任务管理），组件化JS与可视化。
- 业务层（Server/API）：Flask路由层 + Services业务层，负责鉴权、校验、编排与统一响应。
- 数据层（Persistence）：SQLite数据库 + 文件存储（模型/数据集/日志/checkpoint），提供可追踪的元信息与审计。
- 执行层（Execution）：通过conda子进程运行不同模型/训练脚本，隔离依赖环境并通过文件协议交换数据。

### 4.2 功能模块划分

按业务域划分模块如下：
- 用户与权限模块：登录/注册/会话、个人资料、角色授权
- 图数据模块：上传/解析/统计/预设生成、数据集复用
- 模型评估模块：单模型拆解、多模型对比、指标计算、结果导出
- 模型管理模块：模型资产登记、重命名、删除、下载、占用统计
- 训练模块：训练任务创建、执行、曲线采集、checkpoint管理、断点续训
- 管理与运维模块：用户管理、全局任务监控、存储监控与清理、日志查询与CSV导出

### 4.3 数据层总体设计（SQLite）

为满足可追踪性与审计要求，建议建立以下核心实体（详见附录数据字典）：  
- `users`：用户基础信息（邮箱、头像、密码hash、角色、状态等）
- `sessions`（可选）：会话token（或仅使用JWT/无状态token）
- `op_logs`：操作日志（时间、用户、动作、详情、结果、来源等）
- `graphs`（可选）：上传/数据集图的元数据与文件索引
- `datasets`：数据集元数据（名称、格式、大小、所有者、版本、校验）
- `models`：模型资产元数据（类型、展示名、文件路径、所有者、创建方式、指标摘要）
- `dismantle_runs`：拆解评估记录（输入图来源、模型、参数、结果摘要、指标、导出文件索引）
- `train_tasks`：训练任务（状态机、参数json、开始结束时间、当前epoch、异常信息、产物指针）
- `train_metrics`：训练曲线点（task_id, step/epoch, reward/loss/lcc等）
- `checkpoints`：checkpoint索引（task_id, epoch, file_path, is_best, metrics摘要）

### 4.4 关键业务流程（概述）

- 拆解评估流程：图输入→校验与统计→选择模型→调用执行器→指标计算→可视化与导出→记录拆解run与操作日志。
- 训练流程：创建任务→进入队列→后台执行→周期性写入曲线点与checkpoint→完成/失败→登记模型资产→用户/管理员查看任务与曲线。
- 断点续训流程：选择checkpoint→创建“续训任务”（或更新同一任务）→后台从checkpoint恢复→继续写入曲线→完成。
- 存储监控与清理流程：扫描目录统计→展示占用→管理员选择清理策略→执行删除/压缩→写入审计日志。

## 第5章 详细设计

> 说明：本章在不破坏现有原型的前提下，给出新增功能的可落地详细设计，包含接口、数据结构、权限与异常处理策略。实现建议遵循现有 `system/server/services/` 的分层方式。

### 5.1 用户与个人信息模块（Profile）

#### 5.1.1 页面与交互（`profile.html`）
- 基本信息卡：用户名（只读）、角色（只读）、邮箱（可编辑）、头像（上传/预览/删除）。
- 安全设置：修改密码（旧密码/新密码/确认密码）。
- 历史记录Tab：
  - 拆解记录：按时间/模型/图来源筛选，进入详情页查看指标与导出链接。
  - 训练记录：按状态/实验名筛选，进入详情页查看参数、曲线与checkpoint。

#### 5.1.2 典型接口设计（建议）
- `GET /api/profile`：返回当前用户资料（邮箱、头像URL等）。
- `POST /api/profile/update`：更新邮箱/头像；头像采用multipart上传或Base64方案（建议multipart）。
- `POST /api/profile/password`：修改密码（校验旧密码与复杂度）。
- `GET /api/me/history/dismantle`：分页查询个人拆解记录。
- `GET /api/me/history/train`：分页查询个人训练记录。

#### 5.1.3 关键业务规则与边界条件

- 邮箱校验：允许为空；若填写需满足基本格式校验，并可做唯一性约束（可选）。
- 头像文件：限制类型（png/jpg/webp）、大小（例如≤2MB）、存储目录受控；更新头像时可保留旧文件到回收站（可选）。
- 密码策略：长度下限（如≥8）、复杂度（字母+数字等，按项目要求可放宽）；修改密码必须提供旧密码并校验hash。
- 历史记录可见性：普通用户仅可见自身记录；管理员可在管理侧查看全局记录（不通过profile页暴露）。

#### 5.1.3 权限与安全
- 资料修改仅允许本人；管理员可通过管理台修改用户状态，但不直接修改密码（可提供重置密码）。
- 密码存储采用强hash（如bcrypt/argon2），并可加入salt；本报告以“密码hash”表述，不讨论明文存储。

### 5.2 模型管理模块

#### 5.2.1 模型资产抽象

为统一“finder模型目录、mind checkpoint文件、baseline算法方法名、训练产物模型文件”，建议抽象为“模型资产（ModelAsset）”：
- `model_type`：finder / mind / baseline / trained（或按项目划分）
- `display_name`：可重命名
- `artifact_path`：文件路径或方法名
- `owner_user_id`：所属用户（baseline可为空或系统用户）
- `source`：imported / trained / builtin
- `metrics_summary`：摘要指标json（可选）
- `size_bytes`：文件大小（用于存储监控）

#### 5.2.2 功能设计
- 列表：按类型、所有者、创建时间过滤；显示占用与指标摘要。
- 重命名：更新DB中的display_name；必要时对文件名进行映射（建议不直接改底层路径，避免破坏引用）。
- 删除：建议采用软删除字段`deleted_at`，并由管理员“清理任务”执行硬删除与回收空间。
- 下载：对文件资产提供鉴权后的`send_file`下载；对baseline方法提供“导出配置json”或禁用下载。

#### 5.2.3 接口清单（建议）

- `GET /api/models/assets`：模型资产列表（支持`type/owner/keyword/with_deleted`过滤，分页）。
- `POST /api/models/assets/<id>/rename`：重命名（请求体：`display_name`）。
- `DELETE /api/models/assets/<id>`：软删除（普通用户仅限自己；管理员可删除任意）。
- `POST /api/models/assets/<id>/restore`：从回收站恢复（管理员或所有者）。
- `GET /api/models/assets/<id>/download`：下载模型文件（baseline类型返回404或返回配置json）。
- `GET /api/models/assets/<id>`：详情（包含指标摘要、训练来源、文件大小、引用情况）。

#### 5.2.4 审计点（必须写入`op_logs`）

- 模型重命名：记录旧名→新名、操作者、目标模型id。
- 模型删除/恢复：记录原因（可选）、影响文件大小、是否硬删除。
- 模型下载：记录下载次数与来源信息（IP/UA，若采集）。

### 5.3 训练任务与记录持久化

#### 5.3.1 训练任务状态机

定义状态集合：`queued`、`running`、`paused`、`stopped`、`failed`、`succeeded`。  
状态迁移约束：
- `queued -> running`
- `running -> paused -> running`
- `running/paused -> stopped`
- `running -> failed/succeeded`
- `failed`通常不可直接转`running`，应通过“断点续训”创建新任务或显式resume。

#### 5.3.2 曲线与指标采集
- 训练循环内以固定周期写入`train_metrics`（epoch、reward、loss、lcc等）。
- 最优模型策略：若启用“仅保存最优模型”，每次刷新最优指标时写入checkpoint并标记`is_best=1`。
- 训练完成时写入`train_tasks.result_summary`与`models.metrics_summary`。

#### 5.3.3 训练任务日志
- 训练日志建议以文件方式保存（便于流式追加与下载），并在DB保存`log_path`。
- 前端通过轮询接口分页拉取日志尾部（或未来使用WebSocket）。

#### 5.3.4 接口清单（建议）

- `POST /api/train/tasks`：创建训练任务（返回`task_id`），请求体包含`exp_name`与`params`。
- `GET /api/train/tasks`：任务列表（普通用户仅自身；管理员可看全局并可过滤`owner/status`）。
- `GET /api/train/tasks/<id>`：任务详情（参数、状态、时间、当前epoch、最优指标摘要、产物指针）。
- `POST /api/train/tasks/<id>/start`：开始执行（或创建即进入queued，由后台自动start）。
- `POST /api/train/tasks/<id>/pause`：暂停。
- `POST /api/train/tasks/<id>/resume`：继续（不改变checkpoint）。
- `POST /api/train/tasks/<id>/stop`：停止。
- `GET /api/train/tasks/<id>/metrics`：训练曲线点查询（支持`from_epoch/to_epoch/limit`）。
- `GET /api/train/tasks/<id>/checkpoints`：checkpoint列表。
- `GET /api/train/tasks/<id>/logs`：训练日志（支持`offset/limit`或“尾部N行”）。

#### 5.3.5 关键实现要点（面向工程落地）

- 幂等性：重复调用`pause/stop`应返回成功但不重复记录副作用（写一次审计即可）。
- 并发与锁：同一任务不应被多个执行器同时运行；建议引入`task_lock`或“running标记+心跳”机制。
- 心跳：后台执行器周期性更新`train_tasks.current_epoch`与`last_heartbeat_at`（可新增字段），便于管理员判定“卡死任务”。
- 失败恢复：当进程异常退出时，状态标为`failed`并记录`error_message`；保留日志与checkpoint以便续训或诊断。

### 5.4 断点续训（Resume）

#### 5.4.1 checkpoint索引与一致性
- 每个checkpoint均需在DB登记：`task_id`、`epoch`、`file_path`、`metrics_snapshot`、`created_at`。
- 清理checkpoint时遵循引用关系：被“续训任务”引用的checkpoint不可直接硬删。

#### 5.4.2 Resume策略
- **策略A（推荐）**：创建新任务`train_tasks`，并记录`resume_from_checkpoint_id`，将续训视为新的实验阶段，曲线可按“分段”展示或在前端拼接展示。
- **策略B**：在同一任务内续训，需处理曲线重复与epoch对齐；工程上更复杂。

#### 5.4.3 续训接口与校验（建议）

- `POST /api/train/tasks/<id>/resume_from_checkpoint`：基于checkpoint创建“续训任务”。请求体：`checkpoint_id`、（可选）`override_params`。
- 校验规则：
  - checkpoint必须属于该用户（或管理员）可见范围；
  - checkpoint对应文件必须存在且hash一致（可选）；
  - 若原任务仍处于`running/paused`，应要求先停止或创建“并行分支任务”（默认禁止并行分支，避免资源冲突）。

### 5.5 存储空间监控与清理

#### 5.5.1 统计维度
- 磁盘总览（系统盘/数据盘，或配置化根目录）。
- 目录占用：`models/`、`datasets/`、`logs/`、`tmp/`、`checkpoints/`等。
- 资产占用排行：按模型/数据集文件大小排行。

#### 5.5.2 清理策略
- 临时文件：按时间TTL清理。
- 软删除资产：管理员触发“回收站清理”进行硬删除与空间回收。
- 无引用文件：对DB中不存在索引的孤儿文件提示管理员处理（谨慎）。

#### 5.5.3 接口清单（建议）

- `GET /api/admin/storage/summary`：磁盘总览与各目录占用统计（管理员）。
- `GET /api/admin/storage/top`：按模型/数据集/日志占用排行（管理员）。
- `POST /api/admin/storage/cleanup`：执行清理（管理员），请求体包含`policy`与参数（如TTL天数、仅清理软删除等）。

### 5.6 数据集管理

#### 5.6.1 数据集元数据
- 名称、描述、标签、上传者、格式（edgelist/json/gml/graphml/pkl）、规模统计（节点/边）、文件大小、hash校验。

#### 5.6.2 与训练/评估复用
- 评估页与训练页均支持从数据集列表选择数据源；后端统一通过`GraphProcessor.load_from_file`加载并校验。

#### 5.6.3 接口清单（建议）

- `POST /api/datasets`：上传数据集（multipart），写入元信息并返回统计（节点/边等）。
- `GET /api/datasets`：数据集列表（分页、按所有者/格式/关键字过滤）。
- `GET /api/datasets/<id>`：数据集详情（含规模、hash、引用信息）。
- `DELETE /api/datasets/<id>`：软删除（所有者或管理员）。
- `GET /api/datasets/<id>/download`：下载数据集文件（所有者或管理员）。

### 5.7 训练任务监控（管理员）

#### 5.7.1 列表视图
- 全局任务列表：按状态、用户、时间过滤；显示当前epoch、耗时、最近心跳时间、异常摘要。

#### 5.7.2 管理动作
- 终止任务：向后台执行器发送停止信号并更新状态。
- 清理产物：对失败任务或过期任务清理checkpoint与日志（需审计）。

#### 5.7.3 管理端接口（建议）

- `GET /api/admin/train/tasks`：全局任务列表（支持过滤、排序）。
- `GET /api/admin/train/tasks/<id>`：任务详情（含owner信息、日志索引、产物占用摘要）。
- `POST /api/admin/train/tasks/<id>/stop`：强制停止。
- `POST /api/admin/train/tasks/<id>/mark_failed`：人工标记失败（需记录原因）。

### 5.8 系统日志CSV导出

#### 5.8.1 CSV字段建议
- `time`、`username`、`action`、`detail`、`success`、`ip`、`user_agent`

#### 5.8.2 导出机制
- 由管理员发起导出，后端根据过滤条件查询`op_logs`并生成CSV流式响应（避免内存占用）。
- 导出行为写入`op_logs`，记录过滤条件摘要与导出条数。

#### 5.8.3 接口清单（建议）

- `GET /api/admin/logs`：日志查询（现有接口可扩展支持过滤/分页）。
- `GET /api/admin/logs/export.csv`：按查询参数导出CSV（管理员）。
  - 查询参数建议：`from`、`to`、`username`、`action`、`success`

### 6.5 测试结果记录与分析（论文写法模板）

> 说明：在答辩/提交时，可将“截图/日志/导出文件样例”作为证据插入本节，形成“过程可验证、结果可复现”的论文材料。以下为推荐写法模板。

#### 6.5.1 接口测试结果摘要
- 鉴权类接口：对未登录、token失效、权限不足三类场景均返回预期错误码（401/403），且前端能正确跳转/提示。
- 评估类接口：对合法图输入返回成功，包含完整字段；对非法输入（空图、文件缺失、格式错误）返回明确错误信息。
- 管理类接口：管理员可完成用户管理与日志查看；普通用户访问被拒绝并记录审计（若启用）。

#### 6.5.2 关键业务流程验证
- 拆解评估端到端：登录→输入图→执行→可视化→导出→（扩展）写入拆解记录与操作日志。
- 训练任务端到端：创建→执行→曲线与日志轮询→checkpoint生成→暂停/续训→完成→模型资产登记。
- 运维端到端：存储统计展示→清理策略执行→审计日志留存；日志CSV导出可在Excel正常打开且不乱码。

#### 6.5.3 可靠性与边界条件分析
- 长任务与超时：评估执行设定5分钟超时可避免服务被无限阻塞；训练任务通过后台执行器避免HTTP阻塞。
- 断点续训一致性：checkpoint文件索引与DB登记一致；文件缺失时能给出明确错误并阻止续训。
- 清理安全性：软删除与回收站策略可降低误删风险，管理员硬删除需二次确认并审计。

#### 6.5.4 性能与可扩展性分析（定性/定量）
- 定性：三层解耦降低模块耦合，新增模型类型仅需增加接口脚本与配置并在模型资产登记即可。
- 定量（可选）：在固定图规模（例如1k/5k/10k节点）下记录评估耗时与内存占用；在训练任务下记录每epoch耗时与写库开销。

## 第7章 总结与展望

本文完成了网络拆解系统（NetworkDismantling）原型的工程实现与论文式系统化描述，并在现状基线之上提出了可落地的功能扩展方案，补齐了用户与审计、模型与训练治理、数据集与存储运维等关键工程要素。未来工作可从以下方面推进：
- 以任务队列（如Celery/RQ）替代本地后台执行器，支持并发与容错。
- 引入WebSocket实现训练/评估进度的实时推送，提升交互体验。
- 在数据层引入更强的权限与配额管理，实现多用户资源隔离与配额限制。
- 增加更丰富的评估指标与报告生成（自动生成实验报告、对比图表与复现实验脚本）。
